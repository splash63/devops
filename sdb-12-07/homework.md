## Задание 1

Master + пассивный slave

Преимущества:

Высокая доступность: slave — «горячий резерв» для быстрого переключения при сбое master.
Автоматическое переключение (Failover) с помощью инструментов типа Pacemaker или MySQL Group Replication.
Защита данных: актуальная копия на slave снижает риск потери информации.
Тестирование без влияния на рабочую систему: можно проверять обновления или откатывать изменения.
Простота управления: легко настроить и мониторить.

Master + несколько slave

Преимущества:

Масштабирование чтения: распределение запросов SELECT между slave разгружает master.
Балансировка нагрузки через HAProxy/ProxySQL — равномерное распределение трафика.
Географическое распределение: размещение slave в разных регионах снижает задержку для пользователей.
Повышенная отказоустойчивость: выход из строя одного slave не останавливает систему.
Вынесение ресурсоёмких задач: отдельные slave для отчётов, аналитики, бэкапов.
Гибкость: разные настройки и конфигурация для разных slave.
Постепенное обновление без остановки системы.

## Задание 2

План горизонтального и вертикального шардинга базы данных

Таблицы:

Пользователи (users): user_id, name, email, phone, registration_date, city_id.

Книги (books): book_id, title, author, genre, price, publication_year.

Магазины (stores): store_id, name, address, city, manager_id.

Принципы построения системы
Вертикальный шардинг (разделение по столбцам/таблицам):

Разделение таблиц по функциональным модулям:

Модуль «Пользователи»: таблица users.

Модуль «Книги»: таблица books.

Модуль «Магазины»: таблица stores.

Каждый модуль размещается на отдельном сервере (или кластере).

Преимущества: упрощение запросов, локализация нагрузки по бизнес‑функциям, возможность подбора оптимальной СУБД для каждого модуля.

Горизонтальный шардинг (разделение данных внутри таблицы):

Для больших таблиц — разделение по диапазону или хешу:

Таблица users: шардинг по city_id (например, города 1–100 → шард 1, 101–200 → шард 2 и т. д.).

Таблица books: шардинг по genre (фантастика → шард A, детективы → шард B и т. д.).

Таблица stores: шардинг по географическому признаку (по регионам или федеральным округам).

Ключевые преимущества: равномерное распределение нагрузки, масштабирование без замены оборудования, локализация сбоев.

Разбивка между базами данных
Shard Cluster «Users»:

Сервер 1 (Shard 1): пользователи городов 1–100.

Сервер 2 (Shard 2): пользователи городов 101–200.

Shard Cluster «Books»:

Сервер 3 (Shard A): книги жанра «Фантастика».

Сервер 4 (Shard B): книги жанра «Детективы».

Сервер 5 (Shard C): остальные жанры.

Shard Cluster «Stores»:

Сервер 6 (Shard X): магазины Центрального ФО.

Сервер 7 (Shard Y): магазины Сибирского ФО.

Мета‑сервер (Config Server): хранит карту шардов — какая группа данных на каком сервере.

Query Router (прокси‑сервер): принимает запросы от приложений, определяет нужный шард и перенаправляет запрос.

Режимы работы серверов
Shard‑серверы (1–7):

Основной режим: обработка запросов на чтение и запись для своего шарда.

Репликация: каждый шард имеет 1–2 реплики (master‑slave) для отказоустойчивости.

Балансировка: при росте нагрузки можно добавить новые шарды.

Query Router:

Режим прокси: маршрутизация запросов на основе карты шардов.

Кэширование: хранит часто используемые карты шардов для ускорения маршрутизации.

Config Server:

Режим key‑value хранилища: хранит и обновляет карту шардов.

Высокая доступность: репликация данных для защиты от сбоев.
