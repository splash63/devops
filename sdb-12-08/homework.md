# Домашнее задание к занятию "`Резервное копирование баз данных`" - `Гайченков Евгений`

## Задание 1

1.1. Восстановление данных в полном объёме за предыдущий день
Для этого сценария подходит ежедневное полное резервное копирование (Full Backup). При полном копировании создаётся точная копия всей базы данных, включая все данные, объекты системных таблиц, настройки и другие элементы. Такая резервная копия позволяет восстановить базу данных до состояния на момент её создания за один шаг, без необходимости использования дополнительных копий.

Выполнять полное копирование в период минимальной пользовательской активности, чтобы минимизировать влияние на работу системы.
Хранить резервные копии на отдельных носителях или в удалённом хранилище, чтобы обеспечить доступ к ним при выходе из строя основного сервера.
Регулярно проверять целостность резервных копий и проводить тестовые восстановления.

1.2. Восстановление данных за час до предполагаемой поломки
Для минимизации потери данных при сбое требуется более частое резервное копирование. Здесь могут подойти два подхода:

Инкрементное резервное копирование (Incremental Backup). После ежедневного полного бэкапа каждый час копируются только те данные, которые изменились с момента последнего бэкапа (полного или инкрементного). Это уменьшает объём хранимых данных и время на копирование, но для восстановления потребуется последовательное применение всех инкрементных копий вместе с последней полной резервной версией.

Непрерывное резервное копирование (Continuous Data Protection). Каждая транзакция или изменение в базе данных немедленно реплицируется на резервный сервер. Это обеспечивает возможность восстановления данных почти в реальном времени, вплоть до нужного момента.

Если выбран инкрементный метод, важно учитывать, что восстановление может занять больше времени из-за необходимости последовательного применения всех копий.
При использовании непрерывного копирования требуется больше ресурсов для репликации и хранения данных, но оно обеспечивает максимальную гибкость в восстановлении.

1.3.* Моментальное переключение на работающую или починенную базу данных
Такой сценарий возможен благодаря технологиям высокой доступности. Для реализации моментального переключения используются следующие подходы:

Репликация в реальном времени. Синхронная или асинхронная репликация изменений на вторичный сервер базы данных обеспечивает наличие актуальной копии данных, готовой к переключению в случае сбоя основного сервера. При синхронной репликации данные на вторичном сервере обновляются одновременно с основным, что минимизирует потерю данных, но может влиять на производительность. Асинхронная репликация менее требовательна к ресурсам, но допускает небольшую задержку в обновлении данных.

Кластеризация баз данных. Группа серверов работает так, что если один из них выходит из строя, другой автоматически берёт на себя его функции. Примеры технологий: Microsoft SQL Server Always On Availability Groups, Oracle Real Application Clusters (RAC) и другие.

Отказоустойчивые системы (fault-tolerant systems). Системы, спроектированные так, чтобы при сбое одного компонента другой компонент мгновенно заменял его без потери работы.

Системы распределённых баз данных. Данные распределяются между несколькими узлами в разных географических локациях. Это обеспечивает географическое распределение рисков и возможность переключения на работающий узел при сбое другого.

## Задание 2

2.1. Пример команды для резервного копирования (pg_dump)
```bash
pg_dump <параметры> <имя_базы> > <файл_для_сохранения_копии>
```
Пример создания резервной копии базы данных mydb в файл mydb.dump:

```bash
pg_dump -U postgres mydb > mydb.dump
```
Пример команды для восстановления (pg_restore)
```bash
pg_restore -U postgres -Ft -d mydb < mydb.dump
```
2.1.* Автоматизировать резервное копирование можно с помощью планировщика задач, например, cron в Unix-подобных системах.

3.1. Перед началом необходимо включить бинарное логирование в конфигурации MySQL, установив параметр log_bin.

```bash
mysqldump --flush-logs --delete-master-logs --single-transaction --all-databases | gzip > /var/backups/mysql/$(date +%d-%m-%Y_%H-%M-%S)-inc.gz
```
3.1.* Репликация предпочтительнее резервного копирования в следующих случаях:

Требуется высокая доступность данных. При отказе основного сервера можно быстро переключиться на реплику, минимизируя время простоя. Репликация обеспечивает непрерывность бизнес-процессов.

Необходимо распределить нагрузку. Реплики можно использовать для операций чтения, что снижает нагрузку на основной сервер. Например, ресурсоёмкие отчёты можно направлять на реплику.

Географическое распределение данных. Репликация позволяет размещать копии данных в разных регионах, сокращая задержки для пользователей и повышая устойчивость к региональным сбоям.

Аналитика в реальном времени. Репликация на уровне CDC (Change Data Capture) позволяет потоково доставлять данные в очереди сообщений и аналитические витрины, обновляя дашборды и модели машинного обучения с минимальной задержкой.
